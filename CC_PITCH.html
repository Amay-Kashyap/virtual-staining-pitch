The Problem · Scale & Urgency
2 days

More cancer surgeries. Too few pathologists. The bottleneck is getting worse.

Every new biopsy and breast-conserving surgery adds to a pathology queue that already has too few specialists. Today, tissue still must be fixed, sectioned, chemically stained, and reviewed before a pathologist can assess whether the tumour margin is clear. That delay pushes critical decisions past the operating room.

Why now · Cancer case volume is rising while the pathology workforce stays constrained. SwiftDx is built for that imbalance: it helps each pathologist review margins faster, with visual evidence and structured outputs, without removing the pathologist from the loop. FDA issued its first digital pathology Breakthrough Device Designation in April 2025 — the regulatory window is open.

The Problem · Why Existing Approaches Break

The tissue is ready.
The pipeline isn't.

H&E Staining (Status Quo)

Gold standard for tissue assessment — but demands formalin fixation, paraffin embedding, microtome sectioning, and chemical staining. 24–72 hours minimum. Tissue is consumed and cannot be re-processed if the section quality fails.

IHC Add-on Staining

€3,000+ per case in reagent and lab cost layered on top of H&E. Required when H&E alone is ambiguous — meaning even more delay after the 2-day staining wait has already elapsed.

Manual Margin Assessment

A pathologist manually inspects the tumour–healthy tissue boundary under a microscope after staining. 20–40% of breast cancer surgeries require re-excision — a second operation — because the margin was called incorrectly the first time.

The opportunity

Unstained or fluorescent-stained tissue already encodes the structural information H&E reveals. We virtually generate the stain via pixel-level feature amplification and recolouring, then run AI margin classification — in minutes, intraoperatively.

Technology & Innovation · Criterion 2
The Solution · End-to-End AI Pipeline

Tissue in.
Clean margins out.

01 Tissue acquisition Surgical biopsy or lumpectomy specimen captured as a whole-slide image (WSI) using a digital pathology scanner. Input: unstained or fluorescent-stained tissue — no H&E processing required at this stage. WSI
02 Virtual H&E Staining Generative model (RL-guided image translation) amplifies tissue micro-features and recolours pixels to produce a virtual H&E-equivalent image. Proof-of-concept: fluorescent → virtual stain. Target: unstained → H&E. Output in minutes vs. 24–72h wet lab. RL / GAN
03 LVLM Margin Assessment Large vision-language model with a 2D residual unit architecture (MONAI framework) analyses the boundary between tumour and healthy tissue in the virtually stained WSI. Outputs: margin status (clear / close / involved) + heatmap at >90% precision. MONAI ResUnit
04 Structured clinical report Margin status, confidence score, annotated boundary overlay, and SSIM-validated stain quality metric output as HL7 FHIR-compatible JSON for EMR integration. FHIR Output
05 Intraoperative decision Surgeon reviews AI margin report while patient is still on the table. Immediate re-excision if margin is involved — eliminating the need for a separate re-operation entirely. Surgical

Why RL + generative model
RL-guided image translation iteratively optimises pixel recolouring against H&E ground-truth rewards — preserving histological structure (nuclei, stroma, glands) better than supervised CNNs alone.
PoC → Clinical path
Current model uses fluorescent staining as input (available dataset). Final clinical model targets paired unstained/H&E WSI datasets — scarce but being curated via hospital IRB partnerships.

2D residual unit within the MONAI medical imaging framework. Pre-trained on large pathology corpora, fine-tuned on breast cancer margin assessment — classifies tumour boundary with >90% precision.

Risks · Current Constraints
Current Drawbacks · What Is True Today

Promising model.
Limited data.
Clear next bottleneck.

The main drawback right now is dataset access. We are not limited by the idea; we are limited by how much clinically relevant paired data we can train and validate on.

Dataset availability

We currently do not have enough paired unstained tissue and final H&E ground truth data. Because of that, the present model uses fluorescent dye input rather than full unstained-to-H&E staining, since fluorescent data is the only usable dataset currently available.

Model maturity

The proof of concept is directionally strong, but it still needs broader case diversity, scanner diversity, and pathology-reviewed labels before we can claim robust cross-site generalisation.

Clinical readiness

Before deployment, we need IRB-backed data collection, prospective validation, pathologist review, failure-mode analysis, and a documented regulatory pathway. This supports pathologists; it does not replace their judgment.

Future Plan · Execution
Future Plan · Comprehensive Buildout

Partner first.
Build data.
Scale on cloud.

Phase 1
Tie up with a strong hospital or university pathology department and secure IRB approval. First milestone: collect a clean paired dataset of unstained or fresh tissue images, fluorescent images where available, final H&E slides, and pathologist-reviewed margin labels.
Phase 2
Build the data engine: de-identification, annotation workflow, scanner metadata capture, quality checks, and a reproducible dataset split for training, validation, and holdout testing.
Phase 3
Train the virtual staining and LVLM boundary models on the new paired dataset, then benchmark stain quality, tumour-boundary localisation, and margin classification against pathologist review.
Phase 4
Run pilot deployments with the pathologist in loop, measuring turnaround time, agreement, workflow adoption, and repeat-surgery reduction potential before regulatory submission.
Cloud deployment plan

Use online cloud services early so the system is trainable, testable, and deployable beyond a local prototype.

AWS hosting
Use AWS S3 for de-identified whole-slide image storage, GPU compute for training, managed containers for inference APIs, and secure access controls for hospital or university collaborators.
MLOps
Version datasets and models, monitor inference latency and drift, log predictions for auditability, and support rollback so pilots can be run safely and reproducibly.
